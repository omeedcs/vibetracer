# vibetracer v2 Design Spec

**Date:** 2026-03-24
**Status:** Approved
**Approach:** Smart Daemon, Thin Client (single binary, two modes)

## Summary

vibetracer v2 is a ground-up rearchitecture that transforms the tool from a single-process TUI into a daemon + viewer model. The daemon is a lightweight background recorder. The TUI is a read-only viewer that reads from the daemon's edit log on disk. Core new features: per-file playheads, command-level operation grouping, multi-agent tracking (multiple Claude Code sessions), explicit restore system, and 19 color theme presets.

## Decisions

- **Daemon model:** Project-scoped. One daemon per project, started explicitly or auto-started by the TUI.
- **Observer model:** The TUI is an inspector. Timeline scrubbing is visual only. Restoring files to disk is an explicit, separate action with confirmation.
- **Multi-agent:** Multiple Claude Code sessions tracked concurrently. Agent identity baked into the data model. Future-proofed for other tools via `tool_type` field.
- **Conflicts:** Restore operations check blast radius config and suggest atomic restore of coupled files. User can override.
- **Themes:** 19 hardcoded presets compiled into the binary. Selected via config, cyclable at runtime.
- **Communication:** The edit log JSONL file is the interface between daemon and TUI. No socket protocol for data transfer. The TUI tails the file via the `notify` crate. The daemon's Unix socket is only for hook messages and control commands (stop/status).

## 1. Data Model

### EditEvent v2

```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
struct EditEvent {
    id: u64,                       // monotonic within session
    ts: i64,                       // unix millis
    file: String,                  // relative path
    kind: EditKind,                // create/modify/delete
    patch: String,                 // unified diff
    before_hash: Option<String>,   // None for Create events
    after_hash: String,
    lines_added: u32,
    lines_removed: u32,

    // Agent identity
    #[serde(default)]
    agent_id: Option<String>,      // unique per Claude Code session (PID or UUID)
    #[serde(default)]
    agent_label: Option<String>,   // human-readable ("claude-1", "claude-2")

    // Operation grouping
    #[serde(default)]
    operation_id: Option<String>,  // groups edits from one Claude command
    #[serde(default)]
    operation_intent: Option<String>, // what the command was trying to do
    #[serde(default)]
    tool_name: Option<String>,     // "Edit", "Write", "Bash", etc.

    // Restore tagging
    #[serde(default)]
    restore_id: Option<u64>,       // if this edit was generated by a restore operation
}
```

All new fields use `#[serde(default)]` so v1 JSONL lines deserialize correctly with `None` values. The `restore_id` field tags edits that were generated by a restore operation, distinguishing them from organic edits on the timeline.

### RestoreEvent

```rust
struct RestoreEvent {
    id: u64,
    ts: i64,
    scope: RestoreScope,
    files_restored: Vec<RestoreFileEntry>,
}

struct RestoreFileEntry {
    path: String,
    from_hash: String,    // hash of file content before restore
    to_hash: String,      // hash of file content after restore (empty string = file deleted)
}

enum RestoreScope {
    /// Restore a single file. `target_edit_id` is the EditEvent.id (not a vec index).
    /// The restore engine scans the edit log for this ID to find the target hash.
    File { path: String, target_edit_id: u64 },
    Operation { operation_id: String },
    AgentRange { agent_id: String, from_ts: i64, to_ts: i64 },
}
```

Stored in `restores.jsonl`, separate from the edit log. `target_edit_id` in `RestoreScope::File` refers to `EditEvent.id` (the monotonic counter), not a playhead index. The restore engine looks up the edit by scanning for `id == target_edit_id` and uses its `after_hash` as the restore target. If `target_edit_id` refers to a `Create` event and the user wants to restore to *before* that event (i.e., file did not exist), the restore engine deletes the file and records `to_hash: ""` in the `RestoreFileEntry`.

### Session metadata v2

```rust
struct SessionMeta {
    id: String,
    project_path: String,
    started_at: i64,
    mode: SessionMode,             // passive | enriched
    #[serde(default)]
    agents: Vec<AgentInfo>,        // default empty vec for v1 compat
}

struct AgentInfo {
    agent_id: String,
    agent_label: String,
    tool_type: String,             // "claude-code" for now
    first_seen: i64,
    last_seen: i64,
    edit_count: u64,
}
```

`agents` uses `#[serde(default)]` so existing v1 `meta.json` files deserialize with an empty agent list.

### On-disk format

Unchanged: append-only JSONL for edits, content-addressed SHA-256 store for snapshots, numbered JSON checkpoints. The format just has more fields per line. All new fields default gracefully for backward compatibility with v1 data.

## 2. Daemon Architecture

### Responsibilities

The daemon does exactly three things:

1. **Watch** -- `notify` crate watches the project directory, debounced
2. **Snapshot** -- Content-addressed store, same as v1
3. **Log** -- Append `EditEvent` to `edits.jsonl`

No analysis. No playback. No rendering.

The daemon also manages checkpoint directory creation as part of session setup (creating `.vibetracer/sessions/<id>/checkpoints/`), but does not create or manage checkpoints themselves -- that is a TUI concern.

### Lifecycle

```
vibetracer daemon start [path]    # start background recorder
vibetracer daemon stop [path]     # stop recorder
vibetracer daemon status [path]   # PID, session ID, uptime, edit count, active agents
```

**On start:**
1. Resolve project path (default: cwd)
2. Check for existing PID file at `.vibetracer/daemon.pid`:
   - If PID file exists, read the PID and check if the process is alive via `kill(pid, 0)` (Unix)
   - If alive: error "daemon already running (PID {pid})"
   - If dead: log warning "cleaning up stale PID file from crashed daemon", delete PID file and socket file, proceed
3. The CLI spawns the daemon as a child process via `std::process::Command` with stdio detached (`Stdio::null()` for stdin/stdout/stderr). This avoids `fork()` hazards with kqueue/inotify file descriptors and thread state.
4. The child process:
   a. Creates new session under `.vibetracer/sessions/<id>/` (including `checkpoints/` dir)
   b. Writes PID file (`.vibetracer/daemon.pid`) containing its PID and session ID
   c. Binds Unix socket at `.vibetracer/daemon.sock`
   d. Starts the `notify` watcher (watcher must start *after* process spawn, not before)
   e. Enters main loop
5. The parent process polls for the PID file to appear (up to 3 seconds, 50ms intervals). If it appears, print "daemon started (PID {pid}, session {id})" and exit. If timeout, print error and exit with code 1.

**On stop:**
1. Read PID from `.vibetracer/daemon.pid`
2. Send a stop message over the Unix socket: `{"type": "control", "command": "stop"}`
3. Wait up to 5 seconds for the daemon to exit (poll PID liveness)
4. If still alive after timeout, send `SIGTERM`
5. Daemon on receiving stop: write session summary, remove PID file and socket, exit

### Threading model

The daemon runs 3 threads:

1. **Main thread:** Lifecycle management, signal handling, clean shutdown coordination
2. **Watcher thread:** `notify` crate callback thread (managed by notify internally), sends file change events to main thread via `std::sync::mpsc`
3. **Socket thread:** `std::os::unix::net::UnixListener` in blocking mode with `set_nonblocking(false)`. Accepts connections, reads newline-delimited JSON messages, dispatches to main thread via `std::sync::mpsc`. Uses a second channel for control messages (stop signal).

The main thread runs a loop: drain the file-change channel, drain the socket message channel, process events, sleep 50ms. No async runtime needed.

### Hook protocol

Newline-delimited JSON over Unix socket. Two message types:

**Hook message (from Claude Code):**
```json
{"type": "hook", "agent_id": "abc123", "operation_id": "op-7", "tool_name": "Edit", "file": "src/main.rs", "intent": "refactor auth middleware"}
```

**Control message (from CLI):**
```json
{"type": "control", "command": "stop"}
{"type": "control", "command": "status"}
```

**Status response (from daemon):**
```json
{"type": "status", "pid": 12345, "session_id": "20260324-164532-a7f3", "uptime_secs": 3600, "edit_count": 142, "agents": [{"agent_id": "abc", "agent_label": "claude-1", "last_seen": 1711270932}]}
```

### Hook-to-edit correlation

The daemon maintains a pending enrichments map:

```rust
pending_enrichments: HashMap<String, Vec<HookPayload>>  // keyed by filename
```

When a hook message arrives for file X, the payload is pushed into `pending_enrichments["X"]`. When a `notify` event arrives for file X, the daemon pops the *oldest* enrichment for that file (FIFO) and attaches its `agent_id`, `operation_id`, `operation_intent`, and `tool_name` to the `EditEvent`. If no enrichment exists, the event gets `None` for all agent/operation fields.

Enrichments older than 5 seconds are discarded on each drain cycle (stale cleanup). This handles cases where a hook fires but the file change never arrives (e.g., the tool ran but didn't actually modify the file).

### Restore coordination

When the TUI performs a restore, it first sends a message to the daemon via the Unix socket:

```json
{"type": "restore_start", "restore_id": 42, "files": ["src/main.rs", "src/config.rs"]}
```

The daemon adds the `restore_id` to a `pending_restores: HashMap<String, u64>` keyed by filename. When file changes for those files arrive from `notify`, the daemon tags the resulting `EditEvent` with `restore_id: Some(42)`. The TUI then writes the files to disk. After writing, the TUI sends:

```json
{"type": "restore_end", "restore_id": 42}
```

The daemon clears the pending restore entries. Pending restores older than 30 seconds are automatically cleared (timeout safety net). This ensures restore-generated edits are distinguishable from organic edits in the edit log.

**Precedence rules:** If both `pending_restores` and `pending_enrichments` match the same file event, `pending_restores` wins -- the edit is tagged with `restore_id` and the hook enrichment for that file is discarded. The restore was explicitly declared by the TUI; the hook enrichment was coincidental. Additionally, the daemon tags only the *first* file change per file per `restore_id` -- subsequent changes to the same file during the restore window are treated as organic edits.

### Auto-start

`vibetracer` (TUI) checks for a running daemon on startup. If none, starts one using the same `std::process::Command` spawn mechanism. Quitting the TUI does not stop the daemon. `Q` (capital) quits TUI and stops daemon.

## 3. TUI as Viewer

### Connection

The TUI reads directly from the daemon's session directory on disk:

1. Read `.vibetracer/daemon.pid` to find the active session ID
2. Load all existing edits from `edits.jsonl`, skipping any malformed trailing line (truncated writes from crashes are non-fatal -- log a warning and skip)
3. Watch `edits.jsonl` for new appended lines using the `notify` crate (same dependency already used for project file watching)
4. The TUI's file tailer uses a `BufReader` that retains its seek position. On `notify` event, it reads new bytes from the current position. Incomplete lines (no trailing newline) are buffered internally until the next read completes them. This handles the case where the daemon is mid-write when the TUI reads.
5. Build `App` state from loaded edits
6. Enter render loop

Live mode vs. replay mode is the same code path. The only difference is whether new lines keep appearing in the edit log.

### State ownership

**Daemon owns:** watcher, snapshot store, edit log writing, agent tracking, hook socket

**TUI owns:** playback state, playheads (global + per-file), analysis engines, rendering, checkpoints (save/load from session's `checkpoints/` dir), restore operations

**Shared read:** The TUI reads from `edits.jsonl` (append-only, safe for concurrent read) and `snapshots/` (content-addressed, immutable once written). The daemon writes to these. No locking needed because the daemon only appends and the TUI only reads.

The TUI never writes to `edits.jsonl`. The daemon never reads playback state.

### TUI lifecycle

```
vibetracer [path]            # auto-start daemon if needed, open TUI
vibetracer --no-daemon       # single-process mode (see below)
```

TUI can disconnect and reconnect. The daemon keeps recording.

### Single-process mode (`--no-daemon`)

For debugging and backward compatibility. The TUI launches an in-process recorder on a background thread using the same `recorder.rs` code that the daemon uses. This means `recorder.rs` must be importable by both the daemon and the TUI. To support this, `recorder.rs` lives in a shared `recorder/` module at the top level (not inside `daemon/`). The import boundary rule is: `tui/` may import from `recorder/` but not from `daemon/`. `daemon/` imports from `recorder/` and adds process management (PID, socket, daemonization).

## 4. Per-File Playheads & Command-Level Grouping

### Per-file playheads

```rust
struct App {
    /// Global playhead -- index into the flat edits list
    global_playhead: usize,

    /// Per-file playheads -- maps filename to the index within
    /// that file's own edit list (not the global index)
    file_playheads: HashMap<String, usize>,

    /// Files whose playhead is detached from the global playhead
    detached_files: HashSet<String>,
}
```

- **Global scrub** (left/right): moves all non-detached files together. The global playhead advances through the flat edit list; each non-detached file's playhead advances to the corresponding position within its own edit history.
- **Per-file scrub** (select track, shift+left/right): detaches that file from global, moves its playhead independently within its own edit list.
- **Re-attach** (`a`): snaps the selected file's playhead back to where it would be given the current global playhead position. Removes from `detached_files`.
- **Visual:** detached files show their playhead in the theme's `accent_purple` color instead of the default `accent_warm`.

### Command-level grouping

Edits grouped by `operation_id`. Two timeline views:

- **Edit view** (default): each cell is one file edit
- **Command view** (`g` to toggle): each cell is one Claude operation spanning all files it touched

```rust
struct OperationGroup {
    operation_id: String,
    agent_id: Option<String>,
    agent_label: Option<String>,
    intent: Option<String>,
    edits: Vec<usize>,         // indices into the flat edits list
    files_touched: Vec<String>,
    ts_start: i64,
    ts_end: i64,
}
```

### Operation group construction

The TUI maintains a `HashMap<String, OperationGroup>` keyed by `operation_id`. When a new edit arrives:

- If `operation_id` is `Some(id)` and `id` matches an existing group: extend that group's `edits` vec, update `ts_end`, add to `files_touched` if not already present
- If `operation_id` is `Some(id)` and no group exists: create a new `OperationGroup`
- If `operation_id` is `None`: create a singleton group with a synthetic ID (`"ungrouped-{edit_id}"`)

Groups are ordered by `ts_start` for timeline display.

## 5. Multi-Agent Tracking

### Agent identification

Multiple Claude Code sessions each send hook messages with their `agent_id` (process PID). The daemon maintains a live registry:

```rust
struct AgentRegistry {
    agents: HashMap<String, AgentInfo>,
    next_label_number: u32,
}
```

Labels auto-assigned in order of appearance ("claude-1", "claude-2", ...). Persisted in `meta.json`. On each hook message, the daemon updates `last_seen` and `edit_count` for the agent, or creates a new entry if unseen.

### TUI features

- **Agent-colored timeline:** Each agent gets a distinct color from the theme's `agent_colors` array (see Section 7)
- **Agent filtering:** `1`-`9` to solo an agent in command view, `m` to mute
- **Agent-scoped restore:** Undo everything a specific agent did in a time range
- **Status bar:** Shows active agents and their idle time

### Passive fallback

Without hooks, all edits get `agent_id: None`. Timeline works, just without attribution. Agent filtering keys (`1`-`9`) are no-ops when no agents are tracked.

## 6. Restore System

### Naming

"Rewind" = scrubbing through the timeline visually. "Restore" = writing files to disk. Clear separation.

### Three scopes

**Restore file:** Single file to a specific edit point. Select track, scrub, press `R`. The restore engine looks up `EditEvent.id` at the current file playhead position, retrieves the `after_hash` from the snapshot store, and writes it to disk. Special case: if restoring to *before* a `Create` event (the file didn't exist), the file is deleted.

**Restore operation:** All files from one Claude command to their pre-operation state. Toggle to command view, scrub, press `R`. For each file in the operation, restore to the `before_hash` of that file's first edit in the operation. Coupled file suggestions via blast radius check.

**Restore agent range:** Everything an agent did in a time window. Filter to agent, select range, press `Shift+R`. Collects all operations by that agent in the range. Restores each file to its state before the earliest operation.

### Restore-daemon coordination

Before writing files, the TUI sends a `restore_start` message to the daemon (see Section 2). This ensures the daemon tags resulting file-change events with the `restore_id`, so restore-generated edits are distinguishable from organic edits on the timeline. After writing, the TUI sends `restore_end`. The TUI filters edits with `restore_id != None` from the timeline by default (toggle with `x` to show them).

### Undo restore

Every restore writes a `RestoreEvent` to `restores.jsonl` capturing pre-restore hashes. Press `u` to undo. The undo reads the most recent `RestoreEvent`, writes the `from_hash` content back for each file. Restores form a stack -- multiple undos supported.

### Safety

- Confirmation dialog before every restore showing file list and line counts
- Warning if target file has been modified on disk since the snapshot (compare current file hash to the `after_hash` of the most recent edit for that file)
- No silent data loss

## 7. Themes

19 compiled-in presets:

**Dark:** dark (default), catppuccin-mocha, catppuccin-macchiato, gruvbox-dark, tokyo-night, tokyo-night-storm, dracula, nord, kanagawa, rose-pine, one-dark, solarized-dark, everforest-dark

**Light:** light, catppuccin-latte, gruvbox-light, solarized-light, rose-pine-dawn, everforest-light

Config:
```toml
[theme]
preset = "tokyo-night"
```

### Theme struct

```rust
pub struct Theme {
    // Structural colors
    pub bg: Color,
    pub fg: Color,
    pub fg_dim: Color,
    pub fg_muted: Color,
    pub separator: Color,

    // Semantic accent colors
    pub accent_warm: Color,
    pub accent_green: Color,
    pub accent_red: Color,
    pub accent_blue: Color,
    pub accent_purple: Color,

    // Timeline bar colors
    pub bar_filled: Color,
    pub bar_empty: Color,

    // Agent identity colors (6 distinct, visually separable colors)
    pub agent_colors: [Color; 6],
}
```

Each of the 19 presets defines all 18 color values. The `agent_colors` are chosen per-theme to be visually distinct from each other and legible against that theme's background.

For 7+ agents (unlikely but handled), colors wrap: agent 7 uses `agent_colors[0]` with brightness reduced by 30% (each RGB component multiplied by 0.7). Agent 8 uses `agent_colors[1]` dimmed, etc. All `agent_colors` values must be `Color::Rgb(r, g, b)` variants (not indexed colors) so that per-component dimming is possible.

Press `t` to cycle themes at runtime. Theme name flashes in status bar for 2 seconds. Config file is not auto-updated -- if you want persistence, edit the TOML.

## 9. CLI Interface

```
vibetracer [path]                          # TUI (auto-starts daemon)
vibetracer --no-daemon [path]              # single-process mode
vibetracer --theme <preset> [path]         # override theme

vibetracer daemon start [path]             # start background recorder
vibetracer daemon stop [path]              # stop recorder
vibetracer daemon status [path]            # show status

vibetracer sessions [path]                 # list past sessions
vibetracer replay <session-id>             # replay past session

vibetracer import [session-id]             # import Claude Code JSONL
vibetracer init [path]                     # auto-detect config

vibetracer restore <file> <edit-id>        # CLI file restore (edit-id is EditEvent.id)
vibetracer restore --operation <op-id>     # CLI operation restore
```

The `vibetracer restore` CLI subcommand runs headlessly (no TUI). It imports from `restore/` and `snapshot/` to look up hashes and write files. It reads the edit log to find the target edit by `EditEvent.id`, connects to the daemon socket to send `restore_start`/`restore_end` for tagging, and writes `restores.jsonl` for undo support.

### TUI Keybindings

| Key | Action |
|-----|--------|
| `space` | Play/pause |
| `left/right` | Global scrub |
| `shift+left/right` | Per-file scrub (detaches from global) |
| `a` | Re-attach detached file to global playhead |
| `g` | Toggle edit view / command view |
| `R` | Restore (context-dependent scope) |
| `shift+R` | Restore agent range (command view only) |
| `u` | Undo last restore |
| `c` | Create checkpoint |
| `x` | Toggle showing restore-generated edits on timeline |
| `s` | Solo track |
| `m` | Mute track |
| `b` | Blast radius panel |
| `i` | Sentinels panel |
| `w` | Watchdog panel |
| `t` | Cycle theme |
| `1-9` | Solo agent (command view only, no-op in edit view) |
| `?` | Help overlay |
| `q` | Quit TUI (daemon keeps running) |
| `Q` | Quit TUI and stop daemon |

Note: v1's `1`-`5` speed control keys are removed. `R` (shift+r) is reassigned from v1's `RewindFile` to the new context-dependent restore.

### Boundaries

- `recorder/` is shared: imported by both `daemon/` and `tui/` (for `--no-daemon` mode)
- `daemon/` imports from `recorder/` but never from `tui/`
- `tui/` imports from `recorder/` (for `--no-daemon`) but never from `daemon/`
- `restore/` is imported by `tui/` and by `main.rs` (for CLI `restore` subcommand)
- `checkpoint/` is imported only by `tui/`
- `analysis/` is imported only by `tui/`
- `snapshot/` is shared: imported by `recorder/`, `restore/`, `tui/`, and `main.rs`

- `src/rewind/` -- replaced by `src/restore/`
- `src/equation/` -- equation lens cut

- `src/splash.rs` -- splash screen cut
- `src/demo.rs` -- demo mode cut
- `src/analysis/schema_diff.rs` -- schema diff cut
